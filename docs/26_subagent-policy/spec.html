Proposal · Daemon API

Unified Delegation

Merge the policy-request framework and the subagent framework into one delegation surface. Both let an agent hand work off and learn the result; today they have different APIs, different defaults, and different completion semantics. This proposal collapses them into a single model with two explicit delivery modes and a single wait/subscribe primitive that supports any- and all-of fan-in.

1 · The two systems today

Both frameworks let an agent delegate work and observe the outcome. The shapes are startlingly similar — but the wiring is independent.

policy request subagent

Policy request

Agent asks the user to approve running a sandbox-escaping shell command.

  • Created via createPolicyRequest(); daemon writes .clawmini/tmp/requests/<id>.json and appends a role: 'policy' chat message.
  • State: Pending → Approved | Rejected. Resolved by user typing /approve ID or /reject ID.
  • On resolution the request is deleted from disk, a system event message is logged, and the daemon kicks a fresh agent turn carrying the script output as the user’s next message.
  • Always async. The agent has no way to block on the decision; it must finish its turn and let the next turn pick up the result.
  • Auto-approved policies short-circuit: executed inline, never reach Pending.

Subagent

Agent delegates a self-contained task to another agent process.

  • Created via subagentSpawn(); daemon writes a SubagentTracker into ChatSettings.subagents[id] and starts an executeSubagent task.
  • State: active → completed | failed.
  • On completion a role: 'subagent_status' message is logged. If the spawn was async, a <notification> message is also appended for the parent.
  • Sync- or async-able. async: false + a follow-up subagentWait(id) blocks the parent for up to 60s.
  • Default depends on depth: root agents get async = true, subagents get async = false.

2 · Gaps and inconsistencies

Capability Policy request Subagent
Async by default yes (only mode) depth-dependent
Block until resolved no subagentWait(id)
Notify on completion yes (always — kicks new turn) opt-in via async: true
Fire-and-forget (no wakeup) no no
Wait for “all of N” no no (loop on each id)
State retained after resolution no — request file deleted on resolve yes — tracker stays in ChatSettings
Single shared event stream no daemon event subagent_status chat message

3 · Proposed model: one Delegation, two deliveries

Introduce a single concept — a Delegation — that abstracts both a policy approval and a subagent. Two kinds, one lifecycle, one event stream, one wait API.

3.1 The shared lifecycle

What “running” means for each kind For subagents, running is the agent loop executing — seconds to hours, may itself create delegations. For policies, it’s the brief window in which the configured shell script runs inside the /approve handler; delivery only governs what happens with the captured output, not whether the script runs.

3.2 Two delivery modes

Every delegation declares — at creation time — how its result reaches the parent agent:

Mode What happens at resolution Use when…
notify Daemon appends a <notification>…</notification> message into the parent’s chat that wakes the next turn (today’s policy-approval behavior, today’s subagent async: true behavior). You want fire-and-wakeup semantics.
manual No notification, no wakeup. State remains observable via delegations list / delegations show; the parent collects results explicitly via delegationWait when (and if) it cares. Fan-out you’ll collect with a subscription, sync block on a single id, or true fire-and-forget — one mode covers all three. The agent decides whether to wait, register a subscription, or just ignore the result.
Why a per-delegation field, not a per-call argument on wait? Because notification has to be decided before the result is known. If a parent spawns 100 subagents and exits, the daemon needs to know whether each completing subagent should append a wakeup message or stay quiet.

3.3 Defaults that preserve today’s behavior

Caller Today Proposed default
Root agent · policy request kicks fresh turn on resolve delivery: 'notify'
Root agent · subagent spawn async: true → notification delivery: 'notify'
Subagent (depth ≥ 1) · subagent spawn async: false → caller polls subagentWait delivery: 'manual'
Subagent · policy request kicks fresh turn for the subagent on resolve delivery: 'manual' (changed — see callout below)

3.4 Auto-approval is part of the model

Both kinds support being created already-approved when their respective rule says so. This is §4 in detail; here it’s enough to know that auto-approval is a property of creation, not a separate API. Policies keep their existing per-definition autoApprove flag in policies.json; subagents get a new sibling subagents rule list in the same file. An auto-approved delegation skips the pending state entirely and goes straight to running / completed.

4 · Auto-approval rules (policies & subagents)

Today policy definitions carry an autoApprove: true flag and subagent spawns are unconditional — any agent can spawn or send to any other agent without user review. The unified model adds approval gating for subagents while leaving the existing policy shape untouched: existing policies.json files keep working with no migration. Two changes:

  • Policies continue to declare auto-approval per definition, exactly as today ("policies": { "name": { ..., "autoApprove": true } }).
  • A new sibling subagents array in the same file holds spawn/send rules: [{ from, to, autoApprove }]. Absent or empty ⇒ today’s permissive behavior is gone, but the built-in $self → $self rule (below) keeps a subagent able to talk to itself.

4.1 The two rule locations

Both live in the same per-agent policies.json resolved by readPoliciesForPath(); they don’t share a schema because they answer different questions.

  • Policies: per-definition autoApprove: boolean on each entry under "policies" — unchanged from today.
  • Subagents: ordered list under "subagents": each entry is { from, to, autoApprove }. from is matched against the spawner’s agent path and to against the target’s. Endpoints are agent directory paths, path prefixes, "*", or "$self". The no-match default is false, so an explicit false is only useful as a carve-out within a broader true region (e.g. overriding a built-in).

4.2 Built-in subagent rules and how to disable them

Subagent rule resolution is fed a list with user rules first, built-ins appended. Built-ins ship in code (mirroring BUILTIN_POLICIES in src/shared/policies.ts) so users don’t need to copy-paste them into every agent’s config: 

// src/shared/approvals.ts
export const BUILTIN_SUBAGENT_RULES: SubagentRule[] = [
  { from: '$self', to: '$self', autoApprove: true },
];

Because user rules are evaluated before built-ins and the matcher is first-match-wins, opting out of a built-in is just a normal carve-out — no special opt-out shape required. To disable self-clone auto-approval: 

// policies.json — opt out of the built-in self-clone rule.
{
  "subagents": [
    { "from": "$self", "to": "$self", "autoApprove": false }
  ]
}

Policies have no built-ins: a policy is auto-approved iff its definition says so.

Why paths, not environment names The environments map is keyed by directory prefixes: an agent belongs to whichever entry has the most-specific path that contains it, so multiple agents can share an environment without the user wanting them to trust each other. Subagent rules are independent of environments — they reference agent directory paths directly, and use the same prefix-matching convention so a single rule can cover a folder of agents.

4.3 Example config

// policies.json — existing policy shape preserved; new `subagents` list is additive.
{
  "policies": {
    "run-host":        { "command": "...", "args": [] },
    "manage-policies": { "command": "...", "autoApprove": true }
  },

  "subagents": [
    // jeeves can hand work off to any agent under agents/coding/
    { "from": "agents/jeeves", "to": "agents/coding", "autoApprove": true },

    // coding agents may collaborate amongst themselves
    { "from": "agents/coding", "to": "agents/coding", "autoApprove": true }

    // anything else (e.g. agents/coding/* → agents/jeeves) still requires /approve;
    // the built-in `$self → $self` rule is appended after this list.
  ]
}

4.4 Resolution

On every createPolicyRequest, subagentSpawn, and subagentSend:

  1. Policy requests: look up the named definition; use its autoApprove field (default false). No list traversal — this is exactly today’s logic.
  2. Subagent spawn / send: build the candidate { fromPath, toPath } where paths are agent directories relative to the workspace root. Resolve the rule list: user rules (in the order written in policies.json) followed by BUILTIN_SUBAGENT_RULES. Walk the list; a rule matches iff its from covers the candidate’s fromPath AND its to covers toPath. A field covers a value if it is:
    • exactly that value, or
    • a parent directory of it (path-prefix match), or
    • "*" (any value), or
    • "$self" (matches only when the path equals the spawner’s).
    First matching rule wins. Use its autoApprove. Rationale: users read top-to-bottom and expect the first rule they wrote to apply — no implicit re-sorting by specificity. To carve out an exception, place the more-specific rule before the broader one. No match → default false (require user approval).

The check fires on both subagentSpawn and subagentSend — sending a follow-up prompt across an agent boundary is just as sensitive as the initial spawn.

Why per-message approval, not per-edge Approval rules govern information flow, not just code execution. A trust boundary between agents isn’t only about what runs — it’s about what data crosses. If coder-1 is trusted with private source code and jeeves is trusted with network access, an unconstrained channel between them is an exfiltration path: the user approves a single benign-looking spawn and from then on every subagentSend can leak code to the network agent without further review. Gating every subagentSpawn and subagentSend preserves the user’s ability to vet what crosses the boundary. Users who consider an edge unconditionally safe can write a rule with autoApprove: true and skip the prompts; absent that, every message is reviewable.
Caller Target With config above
agents/jeeves agents/jeeves auto-approve (built-in self-clone)
agents/jeeves agents/coding/coder-1 auto-approve (jeeves → coding rule, prefix match on to)
agents/coding/coder-1 agents/coding/refactor auto-approve (coding → coding rule)
agents/coding/coder-1 agents/jeeves pending — needs /approve
agents/coding/refactor agents/coding/coder-2 (send) auto-approve (coding → coding rule)
What does an approval prompt for a subagent look like? It mirrors today’s policy-request preview: a role: 'policy'-style chat message containing the spawner agent, target agent, the prompt body, and a /approve ID / /reject ID footer. Approval transitions the delegation from pending → running and the daemon actually starts the subagent. Rejection terminates it without ever running.

Auto-approved delegations of either kind still emit DAEMON_EVENT_DELEGATION_RESOLVED when they finish, so a subscription that includes them resolves cleanly.

5 · API surface

5.1 Creating a delegation

Each kind keeps its own create endpoint — the inputs are genuinely different (policy needs commandName+args+fileMappings; subagent needs a prompt and target agent). Both gain the delivery field. 

// agent-side, via clawmini-lite
client.createPolicyRequest.mutate({
  commandName: 'run-host',
  args: ['ls', '-la'],
  fileMappings: {},
  delivery: 'manual',        // NEW — default 'notify' for main agents
});

client.subagentSpawn.mutate({
  prompt: 'triage one email',
  delivery: 'notify',        // NEW — replaces `async`
});

5.2 The unified wait / subscribe

One endpoint, two return modes:

client.delegationWait.mutate({
  ids: ['A1B', 'C2D', 'E3F'],
  mode: 'all',               // 'any' | 'all'
  return: 'sync',            // 'sync' = block; 'subscribe' = register, fire later
  timeoutMs: 60_000,         // only meaningful when return === 'sync'
});
  • return: 'sync' blocks the calling RPC until mode is satisfied or timeoutMs elapses. Returns the resolved set + still-pending set.
  • return: 'subscribe' registers a daemon-side subscription and returns a subscriptionId immediately. When mode is later satisfied, the daemon appends one <notification> to the chat with the aggregated outputs and discards the subscription.
  • IDs from both kinds may be mixed. Both kinds use the same short 3-char alphanumeric ID format (growing to 4+ chars on collision); IDs are unique within a chat across kinds, so delegationWait takes plain id strings and the daemon looks each up in either store.
Notification suppression — no double wakeup While a delegation is covered by an unfired subscription or a pending sync wait, its own delivery: 'notify' behavior is suppressed. The covering observer takes the result inline (sync) or aggregates it into one subscription wakeup (subscribe). When the observer resolves, the underlying delegation is already consumed — no separate <notification> is emitted for it. This means it’s safe to mix notify delegations with a subscription; you will get exactly one wakeup, not N+1.

Unsubscribing

client.delegationUnsubscribe.mutate({ subscriptionId });

Unsubscribing discards a subscription without firing a notification. Member delegations are unaffected and continue running — if any of them were created with delivery: 'notify', suppression lifts and they resume normal notify behavior. Use the CLI form delegations unsubscribe <subscriptionId>.

subagentWait(id) stays as a thin compatibility wrapper around delegationWait({ids: [id], mode: 'any', return: 'sync'}) for one release, then is removed.

5.3 Valid combinations

Two axes (delivery on creation, {return, mode} on the wait/subscribe observer) cross-product to a small matrix. All combinations are legal; the table flags which are recommended and which are confusing enough to avoid.

delivery Observer mode Behavior Verdict
notify (none) n/a One wakeup per delegation as each resolves. recommended (main-agent default)
manual any any No wakeup unless an observer is registered. With sync the RPC blocks; with subscribe one aggregated wakeup fires when mode is satisfied. (none) is fire-and-forget — result observable via delegations list / show. recommended for fan-in & subagents
notify sync any Block; covered ids’ notify is suppressed for the duration. On timeout, still-pending ids resume notify. works, but unusual — if you’re blocking you probably wanted manual.
notify subscribe all One aggregated wakeup; member notifies suppressed for the subscription’s lifetime. fine — suppression prevents double-wakeup
notify subscribe any Subscription fires on first resolve; survivors revert to notify, so N−1 individual wakeups follow. discouraged — for exactly one wakeup, switch members to manual.

Suppression rule, in one sentence: a delegation covered by an unfired subscription or pending sync wait does not emit its own notify; once the observer resolves or cancels, suppression ends and remaining-pending ids revert to their declared delivery.

5.4 Listing & introspection

client.delegationList.query({
  state: 'pending' | 'running' | 'resolved' | 'all',
  kind: 'policy' | 'subagent' | undefined,

policiesList / subagentList remain as filtered views.

5.5 Agent CLI: clawmini-lite

Agents talk to the daemon via clawmini-lite on their $PATH; the binary is a thin tRPC client that authenticates with CLAW_API_TOKEN. This is the surface an agent actually sees, so the spec is grounded here.

Design principle: kind-specific commands create things; delegations observes and manages them. If you have an id, reach for delegations regardless of kind.

Policy commands (kind-specific)

Command Purpose & flags
requests list List policy definitions (not requests). Unchanged.
requests show <name> Print one policy definition + its script. Unchanged.
request <cmd> [-f name=path …] [--delivery manual|notify] -- <args…> Create a policy request. New --delivery flag (default 'notify' for main agents, 'manual' for subagents). Prints the new id and a hint pointing at delegations wait / delegations show; auto-approved policies still execute inline and print stdout/stderr.

Subagent commands (kind-specific)

Command Purpose & flags
subagents spawn <prompt> [--agent <name>] [--id <id>] [--delivery manual|notify] Create a subagent. New --delivery flag replaces --async (kept as deprecated alias for one release). Now also subject to the agent-path approval check — a denied edge prints state=pending requiresApproval=true. Prints the new id and hints at delegations wait <id> / delegations notify-when <id>.
subagents send <id> --prompt <p> [--delivery …] Send a follow-up to an existing subagent. Now also gated by the same approval rules as spawn.
subagents stop <id> Abort a running subagent without removing its record. Unchanged.
subagents tail <id> [-n N] [--json] Read a subagent’s message history. Unchanged.

subagents wait, subagents list, and subagents delete are removed — use delegations wait, delegations list --kind subagent, and delegations delete.

delegations command group (kind-agnostic) new

Command Purpose & flags
delegations list [--state pending|running|resolved|all] [--kind policy|subagent] [--json] Cross-kind listing. --state defaults to pending,running; --kind filters; --json emits the raw records.
delegations wait <id…> [--all] [--timeout <ms>] [--subscribe] The unified wait. Default behavior blocks the RPC (mode 'any', timeout 60 000 ms). --all waits for every id; --subscribe registers a daemon-side subscription and returns the subscriptionId immediately. Prints a JSON object: { resolved: [{ id, kind, state, … }], pending: [...] }.
delegations notify-when <id…> [--all] Convenience alias for delegations wait … --subscribe. Returns the subscriptionId immediately; the daemon will append one <notification> chat message to the originating session when the condition is met. Persists across /new; cleared on daemon restart.
delegations unsubscribe <subscriptionId> Discard a registered subscription without firing. Members continue running and revert to their declared delivery.
delegations show <id> Print a single delegation’s record (state, delivery, output if resolved). Works for either kind.
delegations delete <id…> Remove a delegation record (subagent: also stops if running). Replaces subagents delete; works for resolved policy requests too. Refuses while a subscription still covers the id; unsubscribe first.

Flag conventions across the new surface: --all sets mode: 'all' (default 'any'); --subscribe sets return: 'subscribe' (default 'sync'); --delivery takes manual | notify.

User-facing slash commands (for context)

Existing slash commands extend cleanly to the unified id space — no new user commands are introduced.

  • /pending — lists pending policy requests and pending subagent approvals together.
  • /approve <id> / /reject <id> [reason] — works for both kinds; the daemon dispatches based on the id’s store.

5.6 Storage and events

Both kinds share one on-disk store and one owning module — the daemon-side DelegationManager. The two pre-existing stores (RequestStore for policy requests, ChatSettings.subagents for subagent trackers) are removed.

Unified record

// src/shared/delegations.ts
export type DelegationKind  = 'policy' | 'subagent';
export type DelegationState = 'pending' | 'running' | 'completed' | 'rejected' | 'failed';
export type DeliveryMode    = 'notify' | 'manual';

interface DelegationBase {
  id: string;                 // 3-char alphanum, unique per chat
  kind: DelegationKind;
  state: DelegationState;
  delivery: DeliveryMode;
  chatId: string;
  agentId: string;            // creator's agent
  parentId?: string;          // creator's subagent id, if any
  createdAt: string;
  resolvedAt?: string;
  rejectionReason?: string;
}

export interface PolicyDelegation extends DelegationBase {
  kind: 'policy';
  commandName: string;
  args: string[];
  fileMappings: Record<string, string>;
  cwd?: string;
  executionResult?: { stdout: string; stderr: string; exitCode: number };
}

export interface SubagentDelegation extends DelegationBase {
  kind: 'subagent';
  targetAgentId: string;
  sessionId: string;
  prompt: string;             // last prompt sent; refreshed on `subagentSend`
}

export type Delegation = PolicyDelegation | SubagentDelegation;

Layout on disk

.clawmini/tmp/delegations/<chatId>/<id>.json           // one file per delegation
.clawmini/tmp/delegations/<chatId>/subscriptions/<subscriptionId>.json

Per-chat directories make listing cheap and align IDs to their natural visibility scope. The parentId field replaces today’s SubagentTracker.parentId for parent/child access checks — the manager exposes assertVisibleTo(callerSubagentId, id) so cross-kind authorization keeps the same semantics today’s subagent-router.ts enforces.

Module shape

Persistence is a thin file-IO layer (DelegationStore); cross-kind coordination lives on DelegationManager: 

// src/daemon/delegation-manager.ts
class DelegationManager {
  // --- creation (called by kind-specific RPCs, not by agents directly) ---
  createPolicy(input: PolicyCreate):     Promise<PolicyDelegation>;
  createSubagent(input: SubagentCreate): Promise<SubagentDelegation>;
  sendToSubagent(id: string, prompt: string): Promise<void>;   // approval-gated

  // --- lifecycle, called by /approve and /reject handlers ---
  approve(id: string, by: 'user' | 'auto'): Promise<void>;      // pending → running
  reject(id: string, reason: string):       Promise<void>;      // pending → rejected
  markResolved(id: string, outcome: ResolvedOutcome): Promise<void>; // emits DELEGATION_RESOLVED

  // --- observation ---
  get(id):    Promise<Delegation | null>;
  list(filter: { chatId; kind?; state?; parentId? }): Promise<Delegation[]>;
  delete(id): Promise<void>;

  // --- waiting / subscriptions ---
  wait(opts: { ids; mode; return: 'sync' | 'subscribe'; timeoutMs?; chatId; sessionId; });
  unsubscribe(subscriptionId: string): Promise<void>;

  // --- daemon lifecycle ---
  wipeAll(): Promise<void>;            // called on daemon start
}

What stays where: per-kind execution (running a policy script; spawning/managing a subagent process) remains in PolicyRequestService and executeSubagent; the manager dispatches into them on approve(). Approval rule evaluation is a pure function in src/shared/approvals.ts. /approve and /reject stay in slash-policies.ts as the user-facing signal — the handler loads the delegation by id and calls manager.approve / manager.reject, regardless of kind.

Lifecycle invariants

  • Records are retained on resolve. A delegation transitions to completed/rejected/failed with resolvedAt and (for policies) executionResult persisted, so a subscription registered after the fact can still observe them.
  • Daemon startup wipes everything. Today active subagents pile up across restarts and completed policy requests are GC’d in a separate code path. The new behavior is manager.wipeAll(): delete the entire .clawmini/tmp/delegations/ tree on daemon start, treating restart as a clean slate. Trade-off: a fan-out in flight when the daemon restarts is gone; the agent will need to reissue. This is intentional — the alternative (zombie clutter) is the worse UX today.
  • New daemon event DAEMON_EVENT_DELEGATION_RESOLVED fires on policy approval/rejection and on subagent terminal status. The wait/subscribe path listens on this single event instead of scraping DAEMON_EVENT_MESSAGE_APPENDED.

6 · Waiting and subscriptions

Subscription binding: chat-scoped, session-stamped A subscription is registered against a chat, not a session or a turn. If the user runs /new between fan-out and fan-in, the subscription still fires — and the notification message is delivered into the session that registered it (recorded at register time as originSessionId), not whatever session is current. This preserves the agent’s view that “the work I started has come back to me.”

6.1 Notification payload (when return: 'subscribe' fires)

Compact by default — a single <notification> message containing the per-id status table and pointers to where full output lives. The full output of each delegation is already addressable via subagents tail <id> and the chat log, so we don’t inline it. 

<notification>
All 100 delegations resolved (mode: 'all').
✓ 97 completed: A1B, A2C, A3D, A4E, ... (97 total)
✗ 3 failed:
   - QF7  (subagent: timeout after 30 min)
   - QG2  (subagent: script-error, exit 137)
   - X9P  (policy: rejected — "not safe to run")
Use `subagents tail <id>` or `requests show <id>` for full output.
</notification>

If a different aggregation shape turns out useful (e.g. the agent literally wants the 100 outputs concatenated), we can add a payload: 'summary' | 'full' field to the wait call later — not in v1.

Synchronous block

// Block this RPC until a single delegation resolves
const r = await client.delegationWait.mutate({
  ids: ['A1B'],
  mode: 'any',
  return: 'sync',
  timeoutMs: 30_000,
});
// r.resolved[0].output.exitCode === 0

Replaces the do { … } while (status === 'active') polling loop in cli/subagent-commands.ts.

Subscription (one wakeup for many)

// Spawn 100 subagents, then register a single subscription
const ids = [];
for (const item of inbox) {
  const r = await client.subagentSpawn.mutate({
    prompt: `triage ${item.id}`,
    delivery: 'manual',
  });
  ids.push(r.id);
}
await client.delegationWait.mutate({
  ids,
  mode: 'all',
  return: 'subscribe',     // register, return immediately
});
// agent exits; gets one wakeup when the last one finishes

8 · Migration & backwards compatibility

Migration is mostly additive at the wire-protocol layer (new fields, deprecated aliases) and a hard cut at the persistence layer (one new store replaces two old ones, daemon-start wipe absorbs any in-flight data). Steps below are roughly the order they should land.

  1. Introduce the unified Delegation schema and DelegationStore. Add src/shared/delegations.ts (types) and src/daemon/delegation-store.ts (per-chat file IO under .clawmini/tmp/delegations/<chatId>/). The store replaces both RequestStore and ChatSettings.subagents.
  2. Add DelegationManager as the cross-kind owner of lifecycle, events, subscriptions, notify-suppression, and the daemon-start wipe. Per-kind execution stays in PolicyRequestService and executeSubagent; the manager dispatches into them on approve().
  3. Add DAEMON_EVENT_DELEGATION_RESOLVED emitted by manager.markResolved for both kinds; the wait/subscribe path listens on this single event instead of scraping DAEMON_EVENT_MESSAGE_APPENDED.
