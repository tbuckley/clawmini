# SPEC: Agent access to prior thread history

## Problem

What the user perceives as a single linear chat with their agent is actually a
sequence of independent agent sessions, sometimes interleaved across different
agent processes. The agent process that runs in response to a user message only
sees the message currently routed to it (plus anything `fetch-pending` returns
for the same session). Earlier turns — including the agent's own prior replies
written by a previous session — are invisible.

This causes confusion when the user replies to something they see in the chat
that the *current* agent session never produced. From the user's point of view
they are continuing the same conversation; from the agent's point of view a
referent has appeared from nowhere.

We need to give the agent a way to read the thread *as the user sees it*.

## Goal

Add a new `clawmini-lite` command that returns a paginated view of the chat,
restricted to messages that would be visible to the user as part of the
agent/user conversation. The agent can request more pages to walk back through
the history when more context is needed.

## Non-goals

- Subagents. They run a single self-contained session and do not need this
  surface. The endpoint will refuse calls made with a subagent token (see
  "Authorization" below).
- Live streaming. This is a pull API, not a subscription. Agents that want to
  observe new messages mid-turn already have `fetch-pending`.
- Tool calls, command logs, policy requests, router system messages, subagent
  status, retries, legacy `log` / `legacy_log` entries, etc. None of these
  are part of the user-visible thread the way the user perceives it as a
  back-and-forth conversation. The visibility rule below is an allowlist,
  not a denylist: any role not explicitly named is filtered out.

## Design

### What counts as "the conversation as the agent should see it"

The user's view and the agent's view of the chat are not symmetric. The user
sees both `role: 'agent'` messages and `displayRole: 'agent'` system messages
(router auto-replies, etc.) as "the agent talking." The agent should *not* see
`displayRole: 'agent'` messages as its own — those were generated by the
system on its behalf and would be confusing if folded into "things I said."

A message is part of the agent-facing thread when **all** of the following
hold:

1. `subagentId` is undefined.
   - Subagent traffic is not part of the user-visible top-level thread.
2. `displayRole !== 'agent'`.
   - Router auto-replies and slash-command system responses opt out via
     `displayRole: 'agent'`. The user sees them; the agent did not author
     them, so they would be confusing if folded into "things I said." This
     clause runs *before* clause 3, so a hypothetical message with
     `role === 'agent'` and `displayRole === 'agent'` is still excluded.
3. One of:
   - `displayRole === 'user'` — adapter slash-command echoes and other
     entries that should look like user input from the agent's perspective.
   - `role === 'user'` — real user messages.
   - `role === 'agent'` — the agent's own replies.

The chat-id scoping (`ctx.tokenPayload.chatId`) is enforced by the caller
when it asks `getMessages` for that chat's jsonl; it is not part of the
in-process predicate.

The session boundary is intentionally ignored — that is the whole point of
the feature.

#### Output role normalization

`displayRole` takes precedence over `role` when both are set. The normalized
output role is:

- `'user'` if `displayRole === 'user'`.
- `'user'` else if `role === 'user'`.
- `'agent'` otherwise. (Given the predicate above, this branch implies
  `role === 'agent'` with no `displayRole` set.)

In practice we never store messages with `role === 'agent'` and
`displayRole === 'user'` together, so the precedence rule is only stated for
completeness — if such a combination ever appeared we would surface it as a
user-side message, matching what the user's UI would render.

### New tRPC endpoint: `getThreadHistory`

Added to `agentRouter` in `src/daemon/api/agent-router.ts`.

Input (zod):

```ts
z.object({
  limit: z.number().int().min(1).max(200).optional(), // default 20
  before: z.string().optional(),                      // message id cursor
})
```

Behavior:

- Reject with `UNAUTHORIZED` if `ctx.tokenPayload` is missing.
- Reject with `BAD_REQUEST` and the message
  `'thread history is not available to subagents'` if
  `ctx.tokenPayload.subagentId` is set. The feature is explicitly not
  available to subagents; they should fail loudly rather than silently see
  a filtered view that excludes their own messages.
- Resolves `chatId` from `ctx.tokenPayload.chatId`.
- Uses the existing `getMessages(chatId, limit, startDir, predicate, before)`
  helper from `src/shared/chats.ts`. That function already supports a backward
  `before` cursor and a predicate, and reads the file backward so we do not
  parse the whole jsonl for typical limits.
- The predicate is the visibility rule from the previous section.

Return shape:

```ts
{
  messages: Array<{
    id: string;
    role: 'user' | 'agent';      // see "Output role normalization" above
    content: string;
    timestamp: string;            // ISO string from BaseMessage
    files?: string[];             // when present on the underlying message
    sessionId?: string;           // raw sessionId for debugging; do not rely on it
  }>;
  hasMore: boolean;               // true if older visible messages exist before
                                  // the oldest returned message
  oldestId: string | undefined;   // id of the oldest returned message; pass as
                                  // `before` to fetch the next page
}
```

Notes:

- `messages` is returned **oldest-first within the page** to match the user's
  reading order, so the agent can render it directly.
- `role` in the output is normalized per "Output role normalization" above.
  Messages with `displayRole === 'agent'` are filtered out upstream and
  never appear at all, so the agent never has to think about `displayRole`.
- `hasMore` is computed by passing `limit + 1` to `getMessages`. The helper
  walks the jsonl backward and reverses, so its return value is oldest-first
  within the matched set. When `limit + 1` matches were found the extra
  (oldest) one sits at index 0; we drop `messages[0]` to leave exactly
  `limit` entries and report `hasMore: true`. If `getMessages` returned
  `≤ limit` matches we return them all and report `hasMore: false`. This
  avoids a second round-trip to answer "is there more?".
- `oldestId` is exposed instead of asking the agent to pluck `messages[0].id`
  itself, because we want to be free to add filtering inside the daemon
  without the agent's pagination breaking. It is the correct cursor to pass
  back as `before` regardless of internal filtering choices.

### CLI surface in `clawmini-lite`

Added to `src/cli/lite.ts`:

```
clawmini-lite history [--limit <n>] [--before <id>] [--json]
```

- `--limit <n>` — defaults to 20, capped at 200 by the server.
- `--before <id>` — message id from a previous response's `oldestId`.
- `--json` — print the JSON envelope as a single line via
  `JSON.stringify(envelope)` (no pretty-printing), terminated by a newline.
  When `messages` is empty and `oldestId` is undefined, the field is omitted
  from the serialized output (standard `JSON.stringify` behavior); callers
  should treat "field absent" and "field === undefined" as equivalent.
  Default (no `--json`) output is human-readable text suitable for direct
  injection into the agent's working context.

Default text rendering, oldest first:

```
[USER] hello
[AGENT] hi there
[USER] can you elaborate?
---
hasMore: true
oldestId: 1730580000000abc
```

The trailing footer is written *only* when `hasMore` is true, so when the
caller has reached the start of the conversation the output ends cleanly with
the messages. (This matters because the script is often piped into another
process.)

The command exits 0 on success; non-zero on auth/network errors. An empty
chat is success with no message lines (and no footer).

### Authorization

Reuses the existing token mechanism — the daemon uses `ctx.tokenPayload.chatId`
exactly the way `logReplyMessage` and `fetchPendingMessages` already do. No
new permission concept is introduced. The subagent guard described above is
the only new auth check.

### Why not just expose `getMessages` directly?

The user-router `getMessages` already returns *everything*, including command
logs, tool calls, subagent traffic, etc. We do not want to leak that surface
through the agent token, both because it is dramatically more data than the
agent needs and because it exposes internal events (raw policy requests,
tool payloads from earlier turns) that are not part of the user's mental
model of the conversation. A purpose-built endpoint with a fixed, narrow
projection is easier to reason about and easier to evolve.

### Files to change

- `src/daemon/api/agent-router.ts` — add `getThreadHistory` procedure, wire
  it into `agentRouter`.
- `src/cli/lite.ts` — add the `history` command using the existing
  `getClient()` helper.
- `src/shared/chats.ts` — no behavior change expected; the existing
  `getMessages(id, limit, startDir, predicate, before)` already covers what
  we need. Verify the `hasMore` computation doesn't need a small helper.

### Risks / open questions

- **Very large `content` fields.** A previous turn could contain a multi-MB
  paste. We do not truncate in this spec — agents that hit context limits
  should reduce `--limit`. If this turns out to bite us in practice, add a
  `--max-content-length` flag in a follow-up rather than baking truncation
  into the wire format now.
- **Files / attachments.** We pass `files` through unchanged (paths relative
  to the workspace, the same form they have on disk). The agent can read
  them via its normal sandboxed filesystem access if needed.
- **Ordering with `displayRole`.** A `system` message with
  `displayRole: 'agent'` is sorted in jsonl append order along with everything
  else, which is what the user sees in their UI, so we get this for free.

## E2E tests

New file: `e2e/agents/lite-history.test.ts`. Modeled on
`e2e/agents/export-lite-func.test.ts` (same `setupSubagentEnv` +
`getAgentCredentials` + `runLite` pattern).

A shared `beforeAll` sets up the environment. Each test uses a fresh chat
(except where noted) so message-buffer state does not bleed between cases.
We seed messages by sending user messages via the regular CLI and by having
the debug-agent emit replies; we also write `displayRole: 'agent'` system
messages directly to disk to cover the router-auto-reply case.

### Tests

1. **Returns the user-visible thread, oldest first**

   Setup: in chat `hist-basic`, send three user messages and let the
   debug-agent reply to each. Confirm the chat ends up with at least three
   `user`/`agent` pairs.

   Run `clawmini-lite history --json` from a fresh agent session in the same
   chat. Assert:
   - exit code 0
   - `messages.length >= 6`
   - `messages` is sorted by `timestamp` ascending
   - the first user message in our seed appears before the last agent reply
   - every entry has `role` ∈ `{user, agent}`
   - no entry has `role: 'tool'`, `'command'`, `'policy'`, `'system'`, or
     `'subagent_status'`
   - no entry has a `subagentId` field

2. **Excludes tool, command, policy, and subagent_status messages**

   Setup: in chat `hist-noise`, run a debug-agent command that writes a tool
   message via `clawmini-lite tool` and that produces a `command` log on
   exit. Optionally seed a `policy` request directly on disk for completeness.

   Run `history --json`. Assert that `messages` contains the user prompt and
   the agent reply but none of the tool / command / policy noise.

3. **Excludes router auto-replies (`displayRole: 'agent'`); includes adapter
   echoes (`displayRole: 'user'`)**

   Setup: in chat `hist-display-roles`, append two messages directly to
   `chat.jsonl`:
   - a `system` message with `event: 'router'`, `displayRole: 'agent'`,
     `content: 'router auto-reply'` (mirroring `slash-policies.ts`).
   - a `system` message with `event: 'other'`, `displayRole: 'user'`,
     `content: 'slash echo'` (mirroring an adapter slash-command echo).
     `event` is required on `SystemMessage` so we pin it to `'other'` here.

   Send a normal user message and let the debug-agent reply, so there is
   real conversation around the synthetic entries.

   Run `history --json`. Assert:
   - the `displayRole: 'agent'` auto-reply is **absent** from `messages`.
   - the `displayRole: 'user'` echo **is** present, with `role: 'user'` in
     the normalized output.
   - the real user message and the agent reply are present in their natural
     order.

4. **Pagination via `--limit` and `--before`**

   Setup: in chat `hist-page`, seed 7 user/agent message pairs (14 visible
   messages total).

   Run `history --json --limit 5`. Assert:
   - `messages.length === 5`
   - `hasMore === true`
   - `oldestId` is set and equals `messages[0].id`
   - the 5 returned messages are the *most recent* 5 visible messages (this
     is the page closest to the live conversation), still oldest-first
     within the page.

   Run `history --json --limit 5 --before <oldestId from previous run>`.
   Assert:
   - `messages.length === 5`
   - `hasMore === true`
   - the page is the previous 5 messages, all with timestamps strictly
     earlier than the previous response's `messages[0].timestamp`.

   Run a third page with `--limit 5 --before <new oldestId>`. Assert:
   - `messages.length === 4` (only 4 remaining visible messages)
   - `hasMore === false`
   - footer is absent in the default text output (covered in test 6).

5. **Subagent token is rejected**

   Setup: spawn a subagent (via the existing subagent flow used in
   `e2e/agents/subagent-lifecycle.test.ts`) and have it run
   `clawmini-lite history` from inside its sandbox.

   Assert:
   - exit code is non-zero
   - stderr contains `'thread history is not available to subagents'` (the
     BAD_REQUEST message defined in "Authorization" above).
   - the subagent did not receive any thread content on stdout

6. **Default text output**

   Run `clawmini-lite history` (no `--json`) against a chat with two user
   messages and one agent reply, with `hasMore: false`.

   Assert:
   - stdout matches:
     ```
     [USER] first
     [AGENT] hi
     [USER] second
     ```
   - stdout does *not* contain `hasMore:` or `oldestId:` (the footer is
     suppressed when there is nothing more to fetch).

   Then seed enough additional messages to push `hasMore` to true and run
   `history --limit 2`. Assert the footer block is present and the
   `oldestId` line matches `messages[0].id` from the equivalent `--json`
   call.

7. **Cross-session visibility (the regression this feature exists to fix)**

   Setup: in chat `hist-cross`, send a user message via the CLI and let the
   debug-agent reply. Capture the agent's `SESSION_ID` from
   `getSessionSettings(...)`. Force the daemon to start a *new* session by
   resetting `SESSION_ID` for the agent (delete the session settings
   `SESSION_ID` env entry) and send another user message — the new session
   replies.

   Have the agent (running in the second session) call
   `clawmini-lite history --json`. Assert that messages from *both* sessions
   are present in the response, in chronological order. This is the user-
   level scenario the feature exists to enable.

8. **Empty chat returns success with no messages**

   Setup: a fresh chat `hist-empty` containing no user/agent traffic. Create
   it via the `chats add` path (or by writing only `command` / `tool` /
   `legacy_log` entries directly to disk through the raw-jsonl helper). Do
   *not* seed it via `getAgentCredentials`, which produces a real user
   message and a debug-agent reply that would be visible to this endpoint.

   Run `history --json`. Assert exit code 0, `messages` is `[]`,
   `hasMore: false`, `oldestId: undefined`.

   Run `history` (no `--json`). Assert exit code 0, stdout is empty (no
   header, no footer).
