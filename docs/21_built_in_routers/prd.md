# Built-in Routers & Session Timeout Fix

## Vision
The goal of this feature is to solidify the concept of "built-in" routers within the Clawmini Daemon. By standardizing internal routers into predefined `GLOBAL_ROUTERS` (which apply to all messages, including jobs and agent replies) and `USER_ROUTERS` (which apply only to direct user inputs), we reduce configuration overhead for users. Additionally, this redesign will fix the existing session timeout bug, ensuring timeout prompts are correctly bound to the specific session they were initiated in, regardless of whether the user creates new sessions in the interim.

## Product/Market Background
Currently, users define routers (like `@clawmini/session-timeout` and `@clawmini/slash-command`) in their settings. These only apply to `handleUserMessage` flows. This architecture poses two issues:
1. Agent messages and scheduled cron jobs do not pass through the router pipeline, limiting their capabilities and skipping critical built-in behaviors.
2. The session timeout feature relies on a single chat-level cron job (`__session_timeout__`). If a user starts a new session, the job executes but fails to trigger the timeout prompt in the context of the previous, now-inactive session.

Moving towards built-in, categorized pipelines makes the system more robust and allows features like session timeouts to track their state more accurately across multiple parallel sessions.

## Use Cases
- **Standardized Processing**: A message generated by an agent or a cron job should run through global state-modifying routers (like `session-timeout`) automatically.
- **Accurate Session Timeouts**: If a user chats in Session A, goes idle, and then starts Session B, the timeout prompt for Session A should correctly identify Session A and only prompt there.
- **Backwards Compatibility**: Existing user configurations that explicitly define `@clawmini/session-timeout` in their `routers` list should continue to work and apply their custom timeouts and prompts without modification.

## Requirements

### 1. Router Pipeline Refactoring
- Define `GLOBAL_ROUTERS` in `src/daemon/routers.ts`. Note that routers should only apply to: (1) messages from the user to an agent, (2) messages from a job to an agent, and (3) messages from an agent to a subagent.
- Define `USER_ROUTERS` that execute only for messages initiated by the user. Any additional routers the user specifies should ONLY apply to user messages to the agent.
- Modify `handleUserMessage` to run `GLOBAL_ROUTERS`, then `USER_ROUTERS` (with built-ins filtered out of user-provided configurations), then custom user routers.
- Modify agent message handling and cron job handling to invoke the `GLOBAL_ROUTERS` pipeline before executing `executeDirectMessage`.
- When filtering user configurations, scan for `@clawmini/session-timeout` to extract any custom configuration (`timeout`, `prompt`) to pass into the global session timeout router. Suppress or ignore other `@clawmini/*` entries in the user-provided list to prevent double execution.

### 2. Session Timeout Fixes
- **Job ID Uniqueness**: Update the `session-timeout` router to append the current session ID to the cron job ID (e.g., `__session_timeout__<sessionId>`).
- **Session Preservation**: Ensure the scheduled job payload includes the `session: { type: 'existing', id: sessionId }` property so that when the cron job fires, it executes within the context of the original session rather than defaulting to the currently active one.
- **Remove Old Jobs**: Ensure the router properly removes existing timeout jobs for the *current* session before adding the new one.

## Security, Privacy, and Accessibility Concerns
- No new security or privacy risks are introduced. All routing happens locally in the daemon.
- Ensuring jobs execute within the proper session context prevents cross-session contamination of context or UI prompts.
